# TikTok → 惠程荣达完整人工发货动作分析

> 分析日期：2026-07-07
>
> 依据：完整操作录屏、真实 USPS 面单、Estrella/Missrola 发货表、`product.xls`
>
> 隐私处理：收件人、地址、完整跟踪号等信息不写入仓库

## 1. 流程结论

这不是一个单纯的“点击发货”动作，而是人在两个网站、两个 Excel 文件和一个 PDF 文件之间充当数据适配器：

```text
TikTok 订单与物流服务
→ TikTok 安排发货
→ 面单 PDF / 跟踪号
→ product.xls 商品映射
→ 惠程荣达批量订单 Excel
→ WMS 订单导入与确认
→ WMS 交接面单上传
→ 最终核验
```

正常路径上的动作都可由程序自动完成。人工只需处理登录安全验证、页面变更、SKU 无确定映射和平台业务错误。

## 3. 面单分析

真实面单是一页约 4×6 英寸的图片型 PDF，没有可提取文本层。可见信息包括：

- 承运服务和服务等级
- 创建日期、始发邮编、路线代码
- 收件信息（敏感，不应额外复制到系统）
- 一维条码、二维条码
- USPS 22 位数字跟踪号
- 仓内/末端路由编码

自动化结论：

1. 面单跟踪号应优先来自 TikTok 安排发货结果和下载上下文。
2. PDF 文件名使用无空格跟踪号；无需为正常流程做 OCR。
3. 图片型 PDF 不能依赖 `pypdf.extract_text()`。
4. 文件名若包含的跟踪号与任务跟踪号不一致，必须阻断。
5. 收件 PII 已存在于面单，WMS Excel 使用占位字段，不应再重复存储和扩散。

## 4. 真实 Excel 规则

基础字段 A–L：

```text
客户单号, 转单号, 运输方式, 国家, 收件公司, 收件人,
州省, 城市, 地址, 电话, 邮编, 重量
```

从 M 列开始，每件商品固定占 5 列：

```text
中文名N, 英文名N, 配货N, 数量N, 申报N
```

已确认规则：

- `配货N` 写完整 SKU，例如 `XCGLM-GLM…`，不是短条码 `GLM…`。
- `数量N` 是实际配货件数。
- `申报N` 在样例中与数量一致。
- 中文名固定为 `头套`，英文名固定为 `wig`。
- 空收件字段使用 `1` 占位；重量样例固定为 `0.1`。
- 主商品、赠品、套装拆分项全部占独立商品槽。
- Missrola 样例出现至少 7 个商品槽，生成器默认预留 10 个。
- 运输方式不能按跟踪号前缀推断：同类前缀在样例中可能对应不同线路。

## 5. product.xls 规则

`product.xls` 有 `分类1`、`分类2`、`产品总` 三张表。`产品总` 是自动化权威数据源：

- 245 条可用映射
- 完整 SKU 唯一
- 产品条码唯一
- 中文产品名称唯一
- 无重复映射

匹配顺序：

```text
完整平台 SKU
→ 短产品条码
→ 标准化后的完整产品名称/变体名
→ 无结果则人工异常队列
```

## 6. 可以跳过的人工动作

以下动作是人机界面的副产品，不应由 RPA 逐像素模仿：

- 打开文件管理器和日期文件夹
- 右键新建/复制 Excel
- 打开 WPS
- 手工横向滚动表格
- 打开 PDF 做肉眼预览
- 在 TikTok、Excel、product.xls 之间复制粘贴

程序应直接处理订单对象、映射表、Excel 文件和 PDF 响应，只保留业务上真正有意义的外部写操作。
